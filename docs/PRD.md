## 一、产品概述

### 1.1 产品愿景

一个**电台新媒体运营部门全员共用**的内容创作平台：从热点发现 → 选题决策 → 新媒体稿撰写（公众号/微博/小红书/抖音）→ 多级审核 → 一键多平台发布 → 数据回流分析，形成完整闭环。广播口播工具作为远期扩展方向。

### 1.2 核心价值主张

| 痛点（现状） | 解法（Insprira 改造后） |
|-------------|----------------------|
| 每天人工刷微博/抖音找热点，耗时 30-60 分钟 | 系统自动聚合 9 平台热点 + 济南本地关键词监控，3 分钟锁定选题 |
| 写稿和审稿靠微信传来传去，版本混乱 | 内置草稿箱 → 审稿流 → 发布队列，全流程在系统内完成 |
| 同一篇内容发公众号/抖音/小红书要分别改写，全靠人肉 | LLM 多平台改写适配，一键生成各平台版本 |
| 不知道内容发出去效果怎么样，凭感觉选题 | 数据看板 + 内容归因，用数据指导选题方向 |
| 济南本地热点难以系统追踪，容易漏报 | 济南本地关键词定制监控 + 竞品媒体动态追踪，实时推送可追选题 |

### 1.3 产品边界（做什么 vs 不做什么）

| ✅ 做 | ❌ 不做（Phase 1-5 不碰） |
|------|---------------------------|
| 新媒体内容创作全流程（公众号/微博/小红书/抖音） | 听众/粉丝互动管理（评论回复等） |
| 多平台发布调度 | 广告投放/流量采买 |
| 济南本地热点监控与选题引擎 | 外部合作方/客户管理 |
| 部门内部协同（审稿/指派/批注） | 财务/营收分析 |
| 数据分析与可视化 | 视频剪辑/音频编辑 |
| 竞品媒体动态追踪 | 广播口播工具（远期扩展） |

## 二、用户画像

### 2.1 角色定义

| 角色 | 职责 | 系统权限 | 人数估计 |
|------|------|----------|----------|
| **管理员** | 系统配置、用户管理、数据授权 | 全部 | 1-2 人 |
| **主编** | 选题决策、稿件终审、发布审批 | 审稿+发布+数据查看 | 1-2 人 |
| **责编** | 选题分配、初审稿件、排期管理 | 审稿+指派+数据查看 | 2-3 人 |
| **新媒体编辑** | 撰稿、多平台改写、素材采集、热点追踪 | 创作+查看 | 3-5 人 |

> **关于「主持人」角色**：广播口播场景（口语化改写、路况简报、片头模板等）移至远期扩展（见[第九章](#九远期扩展广播口播工具)），Phase 1-5 不做。

### 2.2 核心用户场景（新媒体编辑的一天）

```
┌─────────────────────────────────────────────────────┐
│ 08:30  新媒体编辑到岗                                  │
│        → 打开系统「本地热点」面板                       │
│        → 查看昨夜今晨济南关键词监控推送（3 分钟扫完）    │
│        → 筛选 3-5 个可追选题，标记优先级                │
│        → 编辑「选题池」中昨天下班前主编布置的选题         │
│                                                       │
│ 09:30  选题会（系统辅助）                              │
│        → 主编打开选题池，看今日候选选题                  │
│        → 基于本地热点+竞品动态，决定当日主打方向         │
│        → 在系统中指派选题给具体编辑                     │
│                                                       │
│ 10:00  编辑撰稿                                       │
│        → 打开被指派的选题，进入创作模式                   │
│        → 基于热榜素材+知识库素材撰写公众号长文            │
│        → 保存草稿（自动+手动）                          │

│ 11:30  责编初审                                       │
│        → 审核稿件内容质量、事实准确性、违禁词              │
│        → 通过 → 转主编终审 / 驳回 → 附批注退回编辑修改    │
│                                                       │
│ 14:00  主编终审                                       │
│        → 终审通过 → 排入发布队列                        │
│        → 设定推送时间（如明天 7:30 公众号推送）           │

│ 16:00  下午选题                                       │
│        → 再次查看本地热点面板，关注下午新出现的新闻        │
│        → 为第二天准备 2-3 个选题放入选题池               │
│                                                       │
│ 每周一  数据复盘                                      │
│        → 查看上周部门数据看板                           │
│        → 自动生成周报 → 部门会议使用                     │
```

## 四、Phase 1：基础可用

> **目标**：一个新媒体编辑完整跑通「热榜→选题→创作→本地输出」全流程  
> **主力用户**：新媒体编辑（单人）  
> **不做**：任何协同功能、多平台发布  
> **时间**：1-2 周

### 4.1 功能清单

| ID | 功能 | 描述 | 验收标准 |
|----|------|------|----------|
| F1.1 | RedFox 数据接入 | 配置 API Key 后可拉取热榜数据 | 热榜页正常展示抖音/小红书/公众号热榜 |
| F1.2 | LLM 选题生成 | 基于热榜调用 LLM 生成选题建议 | 输入热榜条目，输出 3-5 个选题建议 |
| F1.3 | 基础文章编辑 | 在系统中完成公众号风格长文撰写 | 支持标题+正文输入，保存草稿 |
| F1.4 | 违禁词检测 | 调用 RedFox 违禁词 Skill 检测文案 | 标红违禁词并给出替换建议 |
| F1.5 | gpt-image-2 封面 | 调用 RedFox 生成文章封面图 | 输入文案生成配图 |
| F1.6 | 济南关键词过滤 | 热榜页支持按济南/交通/本地关键词筛选 | 热榜可按自定义关键词组合过滤 |
| F1.7 | 环境配置固化 | .env 配置、依赖安装、启动流程文档化 | README 含完整部署步骤 |

### 4.2 Phase 1 不做

- ❌ 多用户登录 / 权限
- ❌ 审稿流程
- ❌ 多平台发布
- ❌ 数据看板
- ❌ 本地关键词自动监控（手动过滤可做，自动推送留到 Phase 3）

> **新增**：多账号、RBAC 4 级权限、草稿箱、审稿流

### 5.1 功能清单

| ID | 功能 | 描述 | 验收标准 |
|----|------|------|----------|
| F2.1 | 多账号系统 | 从单账号扩展为多用户独立登录 | 每人独立账号密码，互不影响 |
| F2.2 | RBAC 权限 | 管理员/主编/责编/编辑 4 级角色 | 不同角色看到不同菜单和操作按钮 |

| F2.4 | 选题池 | 共享选题列表，支持认领和指派 | 主编可指派选题给编辑，编辑标记进行中/已完成 |

## 六、Phase 3：本地化内容引擎

> **目标**：为新媒体编辑提供济南本地实时热点监控 + 竞品追踪，让编辑不再人肉刷信息，系统自动推送可追选题  
> **主力用户**：新媒体编辑、责编、主编  
> **时间**：3-5 周

### 6.1 核心理念

Phase 3 不是「给主持人做路况简报」，而是**「给新媒体编辑做选题雷达」**。济南交通广播的公众号/微博/小红书/抖音，内容基因是"济南+交通+生活"，需要一套自动化系统帮编辑发现"今天济南发生了什么值得写"。

### 6.2 功能清单

| ID | 功能 | 描述 | 验收标准 |
|----|------|------|----------|
| F3.1 | 济南关键词监控引擎 | 可配置的关键词组合自动轮询 RedFox 热榜/搜索 | 配置"济南+交通/天气/城建/民生"等关键词后，匹配内容自动聚合展示 |
| F3.2 | 监控面板 | 单屏展示所有监控命中条目，按平台/时效/优先级分组 | 类似微博热搜页面，但只显示济南相关，5 分钟扫完 |
| F3.3 | 优先级自动标记 | AI 根据关键词匹配度+内容热度自动标记 🔴必追 🟡可追 🟢备选 | 每天推送的条目有明确优先级，编辑可一键筛选 |
| F3.4 | 选题一键导入 | 监控面板的任何条目 → 一键导入选题池 | 点击"加入选题"，自动带来源链接+原文摘要 |
| F3.5 | 竞品媒体监控 | 追踪同城竞品媒体/同类账号的最新发布 | 支持添加竞品账号（如"山东交通广播"公众号），有新发布自动提醒 |
| F3.6 | 实时推送通知 | 高优先级条目通过系统通知/企微/钉钉推送 | 🔴必追条目出现时即时推送，不需刷新页面 |
| F3.7 | 监控日报 | 每天 8:00 自动汇总昨夜今晨的关键词命中条目 | 编辑上班打开系统，第一眼看到"今晨本地热点汇总" |

## 七、Phase 4：多平台发布

> **目标**：写完一键推送到微信公众号/微博/小红书/抖音  
> **新增**：多平台 API 对接、改写适配、排期发布、状态追踪  
> **时间**：4-8 周

### 7.1 功能清单

| ID | 功能 | 描述 | 验收标准 |
|----|------|------|----------|
| F4.1 | 微信公众号直发 | 对接微信公众平台 API | 草稿→微信后台草稿箱→群发，全流程在系统内完成 |
| F4.2 | 多平台改写适配 | 同一篇稿 → 公众号版/微博版/小红书版/抖音版 | 4 个版本自动生成，编辑微调即可 |
| F4.3 | 发布排期 | 为每篇稿件设定发布时间 | 支持定时发布，日历视图查看排期 |
| F4.4 | 发布队列 | 待发布/已发布/发布失败 状态管理 | 一目了然每篇稿件的发布状态 |
| F4.5 | 发布状态追踪 | 发布后轮询各平台状态 | 成功/失败/审核中 明确标注 |
| F4.6 | 数据回传 | 发布后拉取阅读量/点赞/转发 | 发布 24h/48h/72h 自动回传基础数据 |

### 7.2 平台适配矩阵

| 平台 | 发布方式 | 改写要点 | 优先级 |
|------|----------|----------|--------|
| 微信公众号 | API 直发 | 长文排版，图片尺寸适配 | P0 |
| 微博 | API 直发 | 140 字摘要+话题标签+链接 | P1 |
| 小红书 | 手动+文案导出 | 种草风，emoji，短段落，话题标签 | P1 |
| 抖音 | 手动+文案导出 | 口语化短文案，话题标签，口播脚本 | P1 |
| 视频号 | 手动+文案导出 | 标题党风格 | P2 |

### 7.3 Phase 4 不做

- ❌ 抖音/小红书 API 直发（平台不开放或门槛极高）
- ❌ 付费推广/投流
- ❌ 评论管理

## 八、Phase 5：数据中台

> **目标**：数据驱动运营决策，取代"凭感觉选题"  
> **新增**：全景看板、竞品监控、内容归因、自动报告  
> **时间**：6-12 周

### 8.1 功能清单

| ID | 功能 | 描述 | 验收标准 |
|----|------|------|----------|
| F5.1 | 部门全景看板 | 所有账号关键指标一屏汇总 | 粉丝数/阅读量/互动率/发布频率，可筛选时间范围 |
| F5.2 | 单账号趋势图 | 粉丝增长、阅读量变化曲线 | 支持日/周/月粒度，可导出 |
| F5.3 | 竞品对比 | 竞品媒体账号数据对比 | 并排展示多个竞品账号的关键指标 |
| F5.4 | 内容归因分析 | 哪类内容/哪个时段/哪种标题效果好 | 输出「高互动内容特征」，反向指导选题 |
| F5.5 | 周报自动生成 | 每周一自动生成上周运营周报 | PDF/Markdown，含图表和数据解读 |
| F5.6 | 月报自动生成 | 每月 1 日自动生成上月运营月报 | 含趋势分析+竞品对比+建议 |
| F5.7 | 异常预警 | 数据异常波动自动告警 | 阅读量暴跌/粉丝负增长 即时推送通知 |

### 8.2 Phase 5 不做

- ❌ 财务/营收分析
- ❌ 广告投放数据
- ❌ 自定义 BI 报表引擎（用现成图表库即可）

## 九、远期扩展：广播口播工具

> **触发条件**：Phase 1-5 全部交付且新媒体运营流程稳定运行后  
> **用户**：主持人  
> **不做时间承诺**：以下为方向性描述，具体排期待定

### 9.1 远期功能池

| ID | 功能 | 描述 |
|----|------|------|
| B1 | 口语化改写引擎 | 新媒体稿 → 广播口语稿自动转换 |
| B2 | 口播时长估算 | 按 220 字/分钟标准语速估算播报时长 |
| B3 | 语调标记 | 自动标注语气：急促/轻松/温馨提醒/严肃 |
| B4 | 早高峰简报 | 每天 6:30 自动生成，主持人 3 分钟可读完 |

| B6 | 片头片尾模板库 | 变量渲染：{{主持人}} {{时段}} {{天气}} 等 |

## 十、非功能需求

### 10.1 UI/UX 铁律（东哥标准）

| 原则 | 具体标准 |
|------|----------|
| **拒绝 AI 设计通病** | 禁止大面积留白、禁止关键元素太小 |
| **字体最小限** | 正文 ≥ 16px，标题 ≥ 22px，卡片标题 ≥ 18px |
| **信息密度优先** | "宁可拥挤也不要空旷"，运营后台不需要呼吸感 |
| **禁止蓝紫渐变** | 禁用 AI 味儿配色（紫蓝渐变、毛玻璃、霓虹发光） |
| **色彩体系** | 参考：暖米白底 #F5F0E8、克制的中性灰、功能性强调色 |
| **按钮可点击** | 最小触控区域 40×40px，间距 ≥ 8px |
| **加载有反馈** | 所有异步操作有 loading/骨架屏，禁止白屏等待 |
| **移动端可用** | 监控面板必须在手机上能快速扫读（响应式） |

### 10.2 性能要求

| 指标 | 目标值 |
|------|--------|
| 首页加载 | < 2 秒 |
| 热榜刷新 | < 3 秒 |
| LLM 生成 | < 15 秒（有 loading 动画） |
| 监控面板刷新 | < 5 秒 |
| 并发用户 | Phase 2 起支持 10 人同时在线 |

### 10.3 安全要求

| 要求 | 说明 |
|------|------|
| 密码存储 | scrypt 哈希（继承现有方案） |
| 传输加密 | 内网部署暂不强制 HTTPS，公网须开启 |
| API Key 保护 | 环境变量存储，前端不可见 |
| 知识库凭证 | AES-256-GCM 加密（继承现有方案） |
| 操作审计 | 关键操作（删除/发布/权限变更）全量日志 |

### 10.4 数据诚信

| 原则 | 说明 |
|------|------|
| **禁止伪造数据** | 所有数据标注来源可信度：✅真实来源 / ⚠️估算 / 🔴可能幻觉 |
| **关键数据交叉验证** | 粉丝数/阅读量等关键指标至少两个来源互相印证 |
| **诚实优于自信** | 数据缺失时说"暂无数据"，不补全/不猜测 |

## 十一、技术约束与决策

### 11.1 架构原则

| 原则 | 说明 |
|------|------|
| **渐进式重构** | 不改现有功能，新功能用新架构 |
| **数据层抽象** | RedFox 调用统一走 Adapter，为切换数据源留后路 |
| **SQLite → PostgreSQL** | Phase 1-3 保持 SQLite，Phase 4 迁移 |
| **前端渐进增强** | Phase 1-2 保持 Vanilla JS，Phase 3 起引入 Alpine.js |

### 11.2 关键决策记录

| 决策 | 选择 | 理由 | 日期 |
|------|------|------|------|
| 主力用户定位 | 新媒体运营部门（非主持人） | 新媒体是日常高频场景，广播口播为远期补充 | 2026-06-21 |
| 上游 fork 策略 | jackwong25/insprira，定期 merge upstream | 保留上游更新能力 | 2026-06-21 |
| 数据源 | RedFox API | 唯一提供公开 API 的竞品 | 2026-06-21 |
| 许可证 | AGPL-3.0（继承） | 内网使用合规 | 2026-06-21 |
| 前端框架 | Phase 1-2 保持 Vanilla，Phase 3 Alpine.js | 降低改动风险 | 2026-06-21 |
| 部署方式 | Phase 1-2 本地，Phase 3 起迁内网服务器 | 部门内部使用 | 2026-06-21 |
| 广播口播工具 | 远期扩展，不占 Phase 1-5 资源 | 先解决 80% 日常场景 | 2026-06-21 |

### 11.3 技术栈演进

| 组件 | Phase 1-2 | Phase 3-4 | Phase 5 |
|------|-----------|-----------|---------|
| 后端 | Node.js 单体 | Express Router 拆分 | 微服务雏形 |
| 数据库 | SQLite | SQLite | PostgreSQL |
| 前端 | Vanilla JS + Tailwind | Alpine.js + HTMX | Alpine.js + 图表库 |
| 认证 | scrypt + Session | scrypt + JWT | JWT + RBAC 中间件 |
| 部署 | 本地 npm start | Docker 内网部署 | Docker Compose 生产 |

## 十二、验收标准

### 12.1 阶段验收条件

| 阶段 | 验收条件 |
|------|----------|
| **Phase 1** | 单人完整跑通"热榜→选题→创作→违禁词检测"全流程，产出 1 篇可发布的新媒体稿件 |
| **Phase 2** | 3 人团队完成 1 篇稿件的"编辑撰稿→责编初审→主编终审"全流程 |
| **Phase 3** | 编辑早上一到岗，打开监控面板 5 分钟内锁定 3 个当天可追选题；系统成功推送 1 条竞品动态 |
| **Phase 4** | 1 篇稿件在系统中完成"撰写→多平台改写→定时发布→公众号推送"全链路 |
| **Phase 5** | 每周一自动生成上周运营周报，含图表+解读，替代人工统计 |

### 12.2 全局质量标准

- [ ] 所有用户操作 3 次点击内完成（不包含填表）
- [ ] 无未处理的 JavaScript 错误
- [ ] 违禁词检测 100% 覆盖（已知词库内的词）
- [ ] 关键流程有操作日志可追溯
- [ ] 移动端（375px 宽）监控面板可快速扫读
