## 一、产品概述

### 1.1 产品愿景

一个**电台新媒体部门全员共用**的内容创作平台：从热点发现 → 选题决策 → 广播稿/新媒体稿撰写 → 多级审核 → 一键多平台发布 → 数据回流分析，形成完整闭环。

### 1.2 核心价值主张

| 痛点（现状） | 解法（Insprira 改造后） |
|-------------|----------------------|
| 每天人工刷微博/抖音找热点，耗时 30-60 分钟 | 系统自动聚合 9 平台热点 + 定制关键词监控，3 分钟消化 |
| 写稿和审稿靠微信传来传去，版本混乱 | 内置草稿箱 → 审稿流 → 发布队列，全流程在系统内完成 |
| 同一篇内容发公众号/抖音/小红书要分别改写，全靠人肉 | LLM 多平台改写适配，一键生成各平台版本 |
| 不知道内容发出去效果怎么样，凭感觉选题 | 数据看板 + 内容归因，用数据指导选题方向 |
| 早班主持人只有 30 分钟准备时间 | 6:30 自动推送早高峰简报，3 分钟读完就能上节目 |

### 1.3 产品边界（做什么 vs 不做什么）

| ✅ 做 | ❌ 不做（至少 Phase 1-5 不碰） |
|------|-------------------------------|
| 内容创作全流程工具 | 听众/粉丝互动管理（评论回复等） |
| 多平台发布调度 | 广告投放/流量采买 |
| 部门内部协同 | 外部合作方/客户管理 |
| 数据分析与可视化 | 财务/营收分析 |
| 广播 + 新媒体双场景 | 视频剪辑/音频编辑 |

## 二、用户画像

### 2.1 角色定义

| 角色 | 职责 | 系统权限 | 人数估计 |
|------|------|----------|----------|
| **管理员** | 系统配置、用户管理、数据授权 | 全部 | 1-2 人 |
| **主编** | 选题决策、稿件终审、发布审批 | 审稿+发布+数据查看 | 1-2 人 |
| **责编** | 选题分配、初审稿件、排期管理 | 审稿+指派+数据查看 | 2-3 人 |
| **编辑/作者** | 撰稿、改写、素材采集 | 创作+查看 | 3-5 人 |
| **主持人** | 接收简报、使用广播稿 | 查看+广播模式 | 3-5 人 |

### 2.2 核心用户场景

```
┌─────────────────────────────────────────────────────┐
```

## 四、Phase 1：基础可用

> **目标**：一个人完整跑通「热榜→选题→创作→本地输出」全流程  
> **不做**：任何协同功能、广播定制功能  
> **时间**：1-2 周

### 4.1 功能清单

| ID | 功能 | 描述 | 验收标准 |
|----|------|------|----------|
| F1.1 | RedFox 数据接入 | 配置 API Key 后可拉取热榜数据 | 热榜页正常展示抖音/小红书/公众号热榜 |
| F1.2 | LLM 选题生成 | 基于热榜调用 LLM 生成选题建议 | 输入热榜条目，输出 3-5 个选题建议 |
| F1.3 | 基础文章编辑 | 在系统中完成文章撰写 | 支持标题+正文输入，保存草稿 |
| F1.4 | 违禁词检测 | 调用 RedFox 违禁词 Skill 检测文案 | 标红违禁词并给出替换建议 |
| F1.5 | gpt-image-2 封面 | 调用 RedFox 生成文章封面图 | 输入文案生成配图 |
| F1.6 | 定制热榜关键词 | 配置交通/济南相关关键词筛选 | 热榜可按自定义关键词过滤 |
| F1.7 | 环境配置固化 | .env 配置、依赖安装、启动流程文档化 | README 含完整部署步骤 |

### 4.2 Phase 1 不做

- ❌ 多用户登录
- ❌ 审稿流程
- ❌ 广播稿模式
- ❌ 多平台发布
- ❌ 数据看板

## 五、Phase 2：团队协同

> **目标**：2-5 人团队在同一系统中协作，有审稿流程  
> **新增**：多账号、RBAC、草稿箱、审稿流  
> **时间**：2-4 周

### 5.1 功能清单

| ID | 功能 | 描述 | 验收标准 |
|----|------|------|----------|
| F2.1 | 多账号系统 | 从单账号扩展为多用户独立登录 | 每人独立账号密码，互不影响 |
| F2.2 | RBAC 权限 | 管理员/主编/责编/编辑/主持人 5 级角色 | 不同角色看到不同菜单和操作按钮 |
| F2.3 | 草稿箱 | 自动保存 + 手动保存 + 版本历史 | 断网/关闭不丢稿，可回溯历史版本 |
| F2.4 | 选题池 | 共享选题列表，支持认领和指派 | 主编可指派选题给编辑，编辑标记进行中/已完成 |
| F2.5 | 审稿流程 | 撰稿 → 责编初审 → 主编终审 → 就绪 | 每步有通过/驳回+理由，全流程可追溯 |
| F2.6 | 操作日志 | 记录关键操作（登录/发布/删除/权限变更） | 管理员可查看全部门操作日志 |
| F2.7 | 内容批注 | 审稿时可对具体段落添加批注 | 编辑看到批注后可逐条处理/回复 |

### 5.2 权限矩阵

| 操作 | 管理员 | 主编 | 责编 | 编辑 | 主持人 |
|------|--------|------|------|------|--------|
| 查看热榜 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 创建选题 | ✅ | ✅ | ✅ | ✅ | ❌ |
| 指派选题 | ✅ | ✅ | ✅ | ❌ | ❌ |
| 撰稿 | ✅ | ✅ | ✅ | ✅ | ❌ |
| 初审 | ✅ | ✅ | ✅ | ❌ | ❌ |
| 终审 | ✅ | ✅ | ❌ | ❌ | ❌ |
| 发布 | ✅ | ✅ | ❌ | ❌ | ❌ |
| 查看数据 | ✅ | ✅ | ✅ | ✅ | ❌ |
| 用户管理 | ✅ | ❌ | ❌ | ❌ | ❌ |
| 系统配置 | ✅ | ❌ | ❌ | ❌ | ❌ |

### 5.3 Phase 2 不做

## 六、Phase 3：广播专属

> **目标**：济南交通广播日常工作流的专属工具上线  
> **新增**：广播稿模式、交通监控、高峰简报、模板库  
> **时间**：3-6 周

### 6.1 功能清单

| ID | 功能 | 描述 | 验收标准 |
|----|------|------|----------|
| F3.1 | 口语化改写引擎 | 书面稿 → 广播口语稿自动转换 | 输入文章，输出可直接口播的稿件，口语化率 > 90% |
| F3.2 | 口播时长估算 | 按 220 字/分钟标准语速估算播报时长 | 标注每段「预计口播：15 秒 / 30 秒 / 60 秒」 |
| F3.3 | 语调标记 | 自动标注语气：急促/轻松/温馨提醒/严肃 | 每段稿件顶部有语气标签 |
| F3.4 | 交通关键词监控 | 济南+交通+路况+天气等定制关键词池 | 匹配到的内容自动聚合推送，标记优先级 |
| F3.5 | 早高峰简报 | 每天 6:30 自动生成，3 分钟可读完 | 包含：必报交通(标记优先级) + 天气 + 本地可提 + 竞品动态 |
| F3.6 | 晚高峰简报 | 每天 16:00 自动生成 | 含晚高峰预测 + 当日回顾 + 夜间施工提醒 |
| F3.7 | 片头片尾模板库 | 变量渲染系统：{{主持人}} {{时段}} {{天气}} 等 | 新建模板 → 选择变量 → 一键渲染成完整口播稿 |
| F3.8 | 数字口语化 | "13.7公里" → "十三多公里"，长句自动拆分 | 所有数值类内容自动转换 |
| F3.9 | 竞品电台监控 | 山东交通广播/青岛交通广播等动态追踪 | 竞品发布内容自动采集到监控列表 |

## 七、Phase 4：多平台发布

> **目标**：写完一键推送到微信公众号/抖音/小红书  
> **新增**：多平台 API 对接、改写适配、排期发布、状态追踪  
> **时间**：4-8 周

### 7.1 功能清单

| ID | 功能 | 描述 | 验收标准 |
|----|------|------|----------|
| F4.1 | 微信公众号直发 | 对接微信公众平台 API | 草稿→微信后台草稿箱→群发，全流程在系统内完成 |
| F4.2 | 多平台改写适配 | 同一篇稿 → 公众号版/抖音口播版/小红书图文版 | 3 个版本自动生成，人工微调即可 |
| F4.3 | 发布排期 | 为每篇稿件设定发布时间 | 支持定时发布，日历视图查看排期 |
| F4.4 | 发布队列 | 待发布/已发布/发布失败 状态管理 | 一目了然每篇稿件的发布状态 |
| F4.5 | 发布状态追踪 | 发布后轮询各平台状态 | 成功/失败/审核中 明确标注 |
| F4.6 | 数据回传 | 发布后拉取阅读量/点赞/转发 | 发布 24h/48h/72h 自动回传基础数据 |

### 7.2 平台适配矩阵

| 平台 | 发布方式 | 改写要点 | 优先级 |
|------|----------|----------|--------|
| 微信公众号 | API 直发 | 长文排版，图片尺寸适配 | P0 |
| 抖音 | 手动+文案导出 | 口语化短文案，话题标签 | P1 |
| 小红书 | 手动+文案导出 | 种草风，emoji，短段落 | P1 |
| 微博 | 手动+文案导出 | 140 字摘要+链接 | P2 |
| 视频号 | 手动+文案导出 | 标题党风格 | P2 |

### 7.3 Phase 4 不做

- ❌ 抖音/小红书 API 直发（平台不开放或门槛极高）
- ❌ 付费推广/投流
- ❌ 评论管理

## 八、Phase 5：数据中台

> **目标**：数据驱动运营决策，取代"凭感觉选题"  
> **新增**：全景看板、竞品监控、内容归因、自动报告  
> **时间**：6-12 周

### 8.1 功能清单

| ID | 功能 | 描述 | 验收标准 |
|----|------|------|----------|
| F5.1 | 部门全景看板 | 所有账号关键指标一屏汇总 | 粉丝数/阅读量/互动率/发布频率，可筛选时间范围 |
| F5.2 | 单账号趋势图 | 粉丝增长、阅读量变化曲线 | 支持日/周/月粒度，可导出 |
| F5.3 | 竞品对比 | 竞品电台账号数据对比 | 并排展示多个竞品账号的关键指标 |
| F5.4 | 内容归因分析 | 哪类内容/哪个时段/哪种标题效果好 | 输出「高互动内容特征」，反向指导选题 |
| F5.5 | 周报自动生成 | 每周一自动生成上周运营周报 | PDF/Markdown，含图表和数据解读 |
| F5.6 | 月报自动生成 | 每月 1 日自动生成上月运营月报 | 含趋势分析+竞品对比+建议 |
| F5.7 | 异常预警 | 数据异常波动自动告警 | 阅读量暴跌/粉丝负增长 即时推送通知 |

### 8.2 Phase 5 不做

- ❌ 财务/营收分析
- ❌ 广告投放数据
- ❌ 自定义 BI 报表引擎（用现成图表库即可）

## 九、非功能需求

### 9.1 UI/UX 铁律（东哥标准）

| 原则 | 具体标准 |
|------|----------|
| **拒绝 AI 设计通病** | 禁止大面积留白、禁止关键元素太小 |
| **字体最小限** | 正文 ≥ 16px，标题 ≥ 22px，卡片标题 ≥ 18px |
| **信息密度优先** | "宁可拥挤也不要空旷"，运营后台不需要呼吸感 |
| **禁止蓝紫渐变** | 禁用 AI 味儿配色（紫蓝渐变、毛玻璃、霓虹发光） |
| **色彩体系** | 参考：暖米白底 #F5F0E8、克制的中性灰、功能性强调色 |
| **按钮可点击** | 最小触控区域 40×40px，间距 ≥ 8px |
| **加载有反馈** | 所有异步操作有 loading/骨架屏，禁止白屏等待 |
| **移动端可用** | 早高峰简报必须在手机上 3 分钟读完（响应式） |

### 9.2 性能要求

| 指标 | 目标值 |
|------|--------|
| 首页加载 | < 2 秒 |
| 热榜刷新 | < 3 秒 |
| LLM 生成 | < 15 秒（有 loading 动画） |
| 简报生成 | < 10 秒 |
| 并发用户 | Phase 2 起支持 10 人同时在线 |

### 9.3 安全要求

| 要求 | 说明 |
|------|------|
| 密码存储 | scrypt 哈希（继承现有方案） |
| 传输加密 | 内网部署暂不强制 HTTPS，公网须开启 |
| API Key 保护 | 环境变量存储，前端不可见 |
| 知识库凭证 | AES-256-GCM 加密（继承现有方案） |
| 操作审计 | 关键操作（删除/发布/权限变更）全量日志 |

### 9.4 数据诚信

| 原则 | 说明 |
|------|------|
| **禁止伪造数据** | 所有数据标注来源可信度：✅真实来源 / ⚠️估算 / 🔴可能幻觉 |
| **关键数据交叉验证** | 粉丝数/阅读量等关键指标至少两个来源互相印证 |
| **诚实优于自信** | 数据缺失时说"暂无数据"，不补全/不猜测 |

## 十、技术约束与决策

### 10.1 架构原则

| 原则 | 说明 |
|------|------|
| **渐进式重构** | 不改现有功能，新功能用新架构 |
| **数据层抽象** | RedFox 调用统一走 Adapter，为切换数据源留后路 |
| **SQLite → PostgreSQL** | Phase 1-3 保持 SQLite，Phase 4 迁移 |
| **前端渐进增强** | Phase 1-2 保持 Vanilla JS，Phase 3 起引入 Alpine.js |

### 10.2 关键决策记录

| 决策 | 选择 | 理由 | 日期 |
|------|------|------|------|
| 上游 fork 策略 | jackwong25/insprira，定期 merge upstream | 保留上游更新能力 | 2026-06-21 |
| 数据源 | RedFox API | 唯一提供公开 API 的竞品 | 2026-06-21 |
| 许可证 | AGPL-3.0（继承） | 内网使用合规 | 2026-06-21 |
| 前端框架 | Phase 1-2 保持 Vanilla，Phase 3 Alpine.js | 降低改动风险 | 2026-06-21 |
| 部署方式 | Phase 1-2 本地，Phase 3 起迁内网服务器 | 部门内部使用 | 2026-06-21 |

### 10.3 技术栈演进

| 组件 | Phase 1-2 | Phase 3-4 | Phase 5 |
|------|-----------|-----------|---------|
| 后端 | Node.js 单体 | Express Router 拆分 | 微服务雏形 |
| 数据库 | SQLite | SQLite | PostgreSQL |
| 前端 | Vanilla JS + Tailwind | Alpine.js + HTMX | Alpine.js + 图表库 |
| 认证 | scrypt + Session | scrypt + JWT | JWT + RBAC 中间件 |
| 部署 | 本地 npm start | Docker 内网部署 | Docker Compose 生产 |

## 十一、验收标准

### 11.1 阶段验收条件

| 阶段 | 验收条件 |
|------|----------|
| **Phase 1** | 单人完整跑通 "热榜→选题→创作→违禁词检测" 全流程，产出 1 篇可发布的稿件 |
| **Phase 2** | 3 人团队完成 1 篇稿件的 "编辑撰稿→责编初审→主编终审" 全流程 |
| **Phase 3** | 某天早高峰，主持人 6:30 打开系统，3 分钟内完成简报消化+模板渲染，7:00 顺利上节目 |
| **Phase 4** | 1 篇稿件在系统中完成 "撰写→改写→定时发布→公众号推送" 全链路 |
| **Phase 5** | 每周一自动生成上周运营周报，含图表+解读，替代人工统计 |

### 11.2 全局质量标准

- [ ] 所有用户操作 3 次点击内完成（不包含填表）
- [ ] 无未处理的 JavaScript 错误
- [ ] 违禁词检测 100% 覆盖（已知词库内的词）
- [ ] 关键流程有操作日志可追溯
- [ ] 移动端（375px 宽）早高峰简报可完整阅读
