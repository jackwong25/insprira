# 交广新媒工坊 → 电台新媒体协同创作平台 · 开发路线图

> 📅 制定日期：2026-06-21 · 版本：v0.1 · 目标：从单人工具进化为部门级协同平台

---

## 一、愿景

将 Insprira 从「个人开发者的自媒体工作台」改造为「**济南交通广播新媒体运营部门的内容协同创作与运营中枢**」。

```
现状：1 人 · 本地单机 · 无协作 · 无审批 · 无发布
      ↓
目标：N 人 · 部门协同 · 审稿流 · 一键多平台发布 · 数据看板
```

---

## 二、核心改造方向

### 🏗️ 方向 A：多人协同（从 "我" 到 "我们"）

| 功能 | 说明 | 复杂度 | 优先级 |
|------|------|--------|--------|
| **RBAC 权限系统** | 管理员/编辑/作者/只读，4 级角色 | 🔴 高 | P0 |
| **多人登录** | 从单账号 SQLite → 多账号体系 | 🔴 高 | P0 |
| **内容指派** | 选题指派给具体编辑，跟踪完成状态 | 🟡 中 | P1 |
| **评论/批注** | 稿子上直接批注讨论 | 🟡 中 | P1 |
| **操作日志** | 谁在什么时候改了什么 | 🟢 低 | P2 |

### ✍️ 方向 B：编辑工作流（从 "写了就发" 到 "三审三校"）

| 功能 | 说明 | 复杂度 | 优先级 |
|------|------|--------|--------|
| **草稿箱** | 保存未完成的稿件，支持版本 | 🟡 中 | P0 |
| **审稿流程** | 撰稿 → 责编审核 → 主编终审 → 发布 | 🔴 高 | P1 |
| **发布队列** | 定时发布，多平台排期 | 🔴 高 | P1 |
| **违禁词前置拦截** | 保存时就检测，不等到发布前 | 🟢 低 | P2 |

### 📡 方向 C：电台场景适配（广播人的专属工具）

| 功能 | 说明 | 复杂度 | 优先级 |
|------|------|--------|--------|
| **广播稿模式** | 口语化改写、时长估算（按语速）、片头片尾模板 | 🟡 中 | P0 |
| **交通关键词监控** | 济南+交通+路况+天气等定制话题追踪 | 🟢 低 | P0 |
| **时段热点简报** | 早高峰/晚高峰自动生成简报 | 🟡 中 | P1 |
| **路况信息采集** | 接入交管/高德 API，自动聚合路况素材 | 🟡 中 | P2 |
| **节目单联动** | 对接现有节目编排系统 | 🔴 高 | P3 |

### 🚀 方向 D：多平台发布（从 "写到" 到 "发到"）

| 功能 | 说明 | 复杂度 | 优先级 |
|------|------|--------|--------|
| **微信公众号直发** | 对接微信公众平台 API，一键推送 | 🔴 高 | P1 |
| **多平台改写适配** | 同一篇稿 → 公众号版/抖音版/小红书版 | 🟡 中 | P1 |
| **发布状态追踪** | 各平台发布是否成功、阅读量回传 | 🟡 中 | P2 |
| **定时发布** | 设定时间自动推送各平台 | 🟡 中 | P2 |

### 📊 方向 E：数据中台（从 "感觉" 到 "数据"）

| 功能 | 说明 | 复杂度 | 优先级 |
|------|------|--------|--------|
| **部门数据看板** | 全部门账号数据一屏汇总 | 🟡 中 | P1 |
| **竞品监控** | 竞品电台/账号的内容和数据分析 | 🟡 中 | P1 |
| **内容效果归因** | 哪类内容效果好 → 指导选题方向 | 🔴 高 | P2 |
| **周报/月报自动生成** | 数据 → 图表 → 报告一键导出 | 🟡 中 | P2 |

### 🔧 方向 F：架构重构（从 "能跑" 到 "能维护"）

| 功能 | 说明 | 复杂度 | 优先级 |
|------|------|--------|--------|
| **server.js 拆分** | 单体 → Express 路由 + Service 层 | 🔴 高 | P1 |
| **前端框架化** | Vanilla JS → Vue/Alpine + 组件化 | 🔴 高 | P1 |
| **数据层抽象** | RedFox → Adapter 接口，留切换后路 | 🟡 中 | P1 |
| **测试覆盖** | 核心 API 测试 + 关键流程 E2E | 🟡 中 | P2 |
| **CI/CD** | GitHub Actions 自动测试+部署 | 🟢 低 | P2 |
| **数据库升级** | SQLite → PostgreSQL（多用户并发） | 🔴 高 | P2 |

## 四、技术路线决策

| 决策点 | 推荐 | 理由 |
|--------|------|------|
| **后端框架** | 渐进式引入 Express | 不改现有路由，新功能用 Express Router |
| **前端框架** | Alpine.js + HTMX | 轻量、不颠覆现有架构、东哥上手快 |
| **数据库** | Phase 1-3 保持 SQLite，Phase 4 迁 PostgreSQL | 初期够用，协作增多后需并发支持 |
| **部署** | Phase 1-2 本地，Phase 3 迁内网服务器 | 部门内部使用，无需公网 |
| **认证** | 保持 scrypt + 加 Session/JWT | 内网环境够用 |
| **代码管理** | monorepo（保持单仓库） | 团队小，单仓库管理成本低 |

## 五、风险与对策

| 风险 | 概率 | 影响 | 对策 |
|------|------|------|------|
| RedFox 服务中断 | 中 | 高 | 数据层抽象 + 缓存；备选 TikHub |
| coracoo 停止维护上游 | 高 | 中 | 已 fork，自主维护；逐步降低对上游依赖 |
| AGPL 合规问题 | 低 | 中 | 内网使用不触发条款；如需外网部署需开源 |
| 架构重构引入 bug | 高 | 中 | 渐进式重构，新功能新架构，老功能保持不动 |
| 团队学习成本 | 中 | 低 | 保留现有简单架构，新功能逐步加入 |
